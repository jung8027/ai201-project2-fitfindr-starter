### Tool 1: search_listings

**What it does:**
Loads the full mock listings dataset from disk, filters items by optional price ceiling and size, scores each remaining item by keyword overlap between the description and the listing's title/description/style_tags/category/colors/brand fields, and returns a relevance-sorted list of matching listing dicts (best match first). Items with a score of zero are dropped.

**Input parameters:**
- `description` (str): Keywords describing what the user is looking for (e.g., "vintage graphic tee"). Used to compute a keyword-overlap relevance score against each listing.
- `size` (str | None): Size string to filter by (e.g., "M", "S/M"). Case-insensitive substring match against the listing's size field. Pass `None` to skip size filtering.
- `max_price` (float | None): Maximum price in dollars, inclusive (e.g., 30.0). Any listing priced above this value is removed before scoring. Pass `None` to skip price filtering.

**What it returns:**
A `list[dict]` of matching listing dicts, sorted by relevance score descending (ties broken by price ascending). Returns an empty list if nothing matches — does not raise an exception. Each dict contains:
- `id` (str): Unique listing identifier
- `title` (str): Human-readable item title
- `description` (str): Seller description text
- `category` (str): Clothing category (e.g., "tops", "bottoms")
- `style_tags` (list[str]): Tags like ["vintage", "streetwear"]
- `size` (str): Garment size (e.g., "M", "S/M", "One Size")
- `condition` (str): Item condition (e.g., "good", "excellent")
- `price` (float): Listing price in USD
- `colors` (list[str]): Dominant colors
- `brand` (str | None): Brand name if known
- `platform` (str): Source platform (e.g., "depop", "poshmark")

**What happens if it fails or returns nothing:**
Returns an empty list. The agent checks `if not results` immediately after the call; if the list is empty it sets `session["error"]` to a user-friendly message ("No listings matched your search. Try a broader description or a higher price limit.") and returns the session early — `suggest_outfit` and `create_fit_card` are never called.

### Tool 2: suggest_outfit

**What it does:**
Takes the selected thrifted listing and the user's wardrobe and returns a non-empty string with 1–2 outfit suggestions. If the wardrobe is empty it asks the LLM for general styling advice (silhouette, color palette, vibe) instead of specific pairings. If the wardrobe has items it instructs the LLM to name specific wardrobe pieces by name in the outfit combinations.

**Input parameters:**
- `new_item` (dict): A listing dict (same shape as returned by `search_listings`) representing the thrifted item the user is considering buying.
- `wardrobe` (dict): A wardrobe dict with an `"items"` key whose value is a list of wardrobe item dicts. May be empty (`{"items": []}`) or missing the key entirely — both are handled gracefully.

**What it returns:**
A non-empty string with outfit suggestions. When the wardrobe has items the string contains two bullet-point outfit combinations that name specific wardrobe pieces. When the wardrobe is empty the string contains general styling advice about silhouette, color palette, and what basics pair well.

**What happens if it fails or returns nothing:**
If the wardrobe is empty, the function calls the LLM with a general-styling prompt rather than raising an error. If the LLM call fails or returns an empty string, the function falls back to a hard-coded styling blurb generated by `_fallback_outfit_text(new_item)` — so the return value is always a non-empty string. The agent never needs to check for failure; it always receives a usable suggestion.

---

### Tool 3: create_fit_card

**What it does:**
Takes the outfit suggestion string and the listing dict and uses the LLM to generate a 2–4 sentence Instagram/TikTok-style caption. The caption naturally mentions the item name, price, and platform once each, sounds casual and authentic (like a real OOTD post), and captures the outfit vibe in specific terms. A higher LLM temperature (0.9) is used so the output sounds different each time.

**Input parameters:**
- `outfit` (str): The outfit suggestion string returned by `suggest_outfit()`. If this is empty or whitespace-only the tool returns a fixed error string without calling the LLM.
- `new_item` (dict): The listing dict for the thrifted item, used to supply the item name, price, and platform to the LLM prompt.

**What it returns:**
A 2–4 sentence string suitable for use as a social-media caption. If `outfit` is empty or whitespace-only, returns the fixed string `"Outfit details are missing, so I can't generate a fit card yet."` instead of calling the LLM. If the LLM call fails, falls back to `_fallback_fit_card(new_item)`.

**What happens if it fails or returns nothing:**
Two guard layers: (1) if `outfit` is empty/whitespace the function short-circuits and returns a descriptive error string immediately; (2) if the LLM call succeeds but returns an empty string, `_fallback_fit_card(new_item)` produces a minimal caption from the listing fields alone. An exception is never raised to the caller.

## Planning Loop

**How does your agent decide which tool to call next?**

The planning loop follows a fixed three-step sequence with one conditional early-exit. It does not iterate — each tool is called at most once per session.

1. **Parse the query.** `_parse_query(query)` runs regex patterns over the lowercased query to extract `description`, `size` (from "size M" or "sz M" patterns), and `max_price` (from "under $30" patterns). The cleaned description is stored in `session["parsed"]`.

2. **Call `search_listings`.** The parsed parameters are passed directly. The result is stored in `session["search_results"]`.
   - **Branch A — empty results:** `if not results` is `True`. Set `session["error"] = "No listings matched…"` and `return session` immediately. `suggest_outfit` and `create_fit_card` are never called.
   - **Branch B — results exist:** Continue to step 3.

3. **Select the top item.** `session["selected_item"] = results[0]` (highest relevance score, ties broken by lowest price).

4. **Call `suggest_outfit`.** Pass `session["selected_item"]` and the original `wardrobe` dict. Store the returned string in `session["outfit_suggestion"]`. This call always returns a non-empty string, so no early exit is needed.

5. **Call `create_fit_card`.** Pass `session["outfit_suggestion"]` and `session["selected_item"]`. Store the returned string in `session["fit_card"]`. This call also always returns a non-empty string.

6. **Return session.** The loop is done. The caller inspects `session["error"]` (None on success) and then `session["fit_card"]`, `session["outfit_suggestion"]`, and `session["search_results"]`.

## State Management

**How does information from one tool get passed to the next?**

All state lives in a single `session` dict created by `_new_session(query, wardrobe)` at the start of each `run_agent()` call. The dict is mutable and is passed around by reference within the function:

| Key | Type | Set by | Read by |
|-----|------|--------|---------|
| `query` | str | `_new_session` | `_parse_query` |
| `parsed` | dict (`description`, `size`, `max_price`) | `_parse_query` | `search_listings` call |
| `search_results` | list[dict] | `search_listings` return | step 3 (early-exit check), `selected_item` assignment |
| `selected_item` | dict | step 3 assignment | `suggest_outfit`, `create_fit_card` calls |
| `wardrobe` | dict | `_new_session` | `suggest_outfit` call |
| `outfit_suggestion` | str | `suggest_outfit` return | `create_fit_card` call |
| `fit_card` | str | `create_fit_card` return | returned to caller |
| `error` | str or None | early-exit branch | returned to caller |

No tool receives the session dict directly — each tool is called with explicit arguments extracted from the session. Results are stored back into the session immediately after each call. This keeps the tools stateless and independently testable.

## Error Handling

For each tool, describe the specific failure mode you're handling and what the agent does in response.

| Tool | Failure mode | Agent response |
|------|-------------|----------------|
| search_listings | No results match the query (empty list returned) | Sets `session["error"]` to `"No listings matched your search. Try a broader description or a higher price limit."` and returns the session immediately. The UI displays this message verbatim and offers no further output — the user is prompted to retry with different keywords or a higher price. `suggest_outfit` and `create_fit_card` are never called. |
| suggest_outfit | Wardrobe is empty (`wardrobe["items"]` is `[]` or key is missing) | The tool switches from a wardrobe-pairing prompt to a general-styling prompt: it asks the LLM for two casual styling ideas focused on silhouette, color palette, and basics to pair with. The user sees styling advice phrased as "here are some general styling ideas for this piece" rather than specific wardrobe combinations. If the LLM also fails, `_fallback_outfit_text(new_item)` produces a sentence suggesting neutral layers and one statement accessory — the caption is still generated normally. |
| create_fit_card | `outfit` string is empty or whitespace-only | Returns the fixed string `"Outfit details are missing, so I can't generate a fit card yet."` without calling the LLM. The user sees this message instead of a caption and is implicitly invited to try a query that produces a valid outfit suggestion first. |

## A Complete Interaction (Step by Step)

FitFindr is a thrift-shopping assistant that helps users find secondhand clothing and style it with what they already own. When a user describes what they want, the agent calls `search_listings` to find matching items from the mock dataset; once a candidate item is chosen, it calls `suggest_outfit` to pair it with the user's wardrobe (or give general styling advice if the wardrobe is empty); finally it calls `create_fit_card` to generate a shareable caption. If any tool fails or returns nothing — no search results, an empty wardrobe, or a missing outfit string — the agent surfaces a friendly fallback message rather than crashing or going silent.

**Example user query:** "I'm looking for a vintage graphic tee under $30. I mostly wear baggy jeans and chunky sneakers. What's out there and how would I style it?"

**Step 1 — Parse the query**

`_parse_query` runs regex over the lowercased query. It finds `"under $30"` → `max_price = 30.0`, finds no "size" or "sz" pattern → `size = None`, and strips the price phrase from the text, leaving `description = "i'm looking for a vintage graphic tee i mostly wear baggy jeans and chunky sneakers what's out there and how would i style it"`. These values are stored in `session["parsed"]`.

**Step 2 — Call `search_listings`**

The agent calls `search_listings(description="i'm looking for a vintage graphic tee...", size=None, max_price=30.0)`. The tool loads all 40 listings and drops any item priced above $30. It then scores each remaining item by counting how many tokens from the description appear in the listing's title, description, style_tags, category, colors, and brand fields. The top-scoring items (all tagged "vintage" and "graphic tee") are:

1. `lst_002` — *Y2K Baby Tee — Butterfly Print*, $18.00, size S/M, depop (score: 3+ token matches)
2. `lst_033` — *Vintage Band Tee — Faded Grey*, $19.00, size L, depop
3. `lst_006` — *Graphic Tee — 2003 Tour Bootleg Style*, $24.00, size L, depop

`session["search_results"]` = the full sorted list. `results` is non-empty so the agent does not set `session["error"]` — it continues to step 3.

**Step 3 — Select the top item**

`session["selected_item"] = results[0]` → the *Y2K Baby Tee — Butterfly Print* dict: `{"id": "lst_002", "title": "Y2K Baby Tee — Butterfly Print", "category": "tops", "style_tags": ["y2k", "vintage", "graphic tee", "cottagecore"], "size": "S/M", "condition": "excellent", "price": 18.0, "colors": ["white", "pink", "purple"], "brand": null, "platform": "depop"}`.

**Step 4 — Call `suggest_outfit`**

The agent calls `suggest_outfit(new_item=<lst_002 dict>, wardrobe={"items": [10 items]})`. The wardrobe is non-empty so the tool formats all 10 wardrobe item names into a prompt and asks the LLM for two specific outfit combinations. The wardrobe includes "Baggy straight-leg jeans, dark wash", "Chunky white sneakers", "Vintage black denim jacket", and "Black combat boots". The LLM returns two bullet points, e.g.:

> • Tuck the Y2K Butterfly Print Tee into your Baggy straight-leg jeans (dark wash) and finish with your Chunky white sneakers — clean vintage-streetwear energy.
> • Layer it under your Vintage black denim jacket over Wide-leg khaki trousers and Black combat boots for a Y2K-meets-grunge look.

`session["outfit_suggestion"]` = that string.

**Step 5 — Call `create_fit_card`**

The agent calls `create_fit_card(outfit=<suggestion from step 4>, new_item=<lst_002 dict>)`. The outfit string is non-empty so the tool builds a prompt asking for a 2–4 sentence Instagram/TikTok caption that mentions the item name, $18 price, and depop platform. The LLM returns a caption such as:

> "Found this Y2K Baby Tee — Butterfly Print on depop for only $18 and I'm obsessed. Styled it tucked into baggy dark-wash jeans with chunky white sneakers — effortless vintage energy. thrift finds hitting different lately ✨ #thrifted #y2k #ootd"

`session["fit_card"]` = that string. `session["error"]` remains `None`.

**Final output to user:**

The user sees three things: (1) the ranked list of matching listings — *Y2K Baby Tee* at $18, *Vintage Band Tee* at $19, *Graphic Tee — 2003 Tour Bootleg Style* at $24 — each showing title, price, condition, size, and platform; (2) two bullet-point outfit ideas that name their specific wardrobe pieces by name; and (3) a ready-to-copy social-media caption they can post directly.
